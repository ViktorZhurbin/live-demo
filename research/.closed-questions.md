## Static preview + interaction-gated `Core`

**The idea.** `<code src>` demos are real files in the site's repo, so the site bundler
can already build them. Inject a normal `import Demo from './snippets/Basic.tsx'` into
the page, server-render `<Demo />` into the preview pane, and don't mount `Core` (or
the compiler) until someone actually interacts with the editor.

**Why it's the biggest remaining idea.** `CodeRunner` lives inside `Core`, which is
behind the `React.lazy` boundary in `lazy.tsx`. Never mounting `Core` means never
loading CodeMirror _or_ Babel — the full ~1.02 MB of demo-only weight, ~87% of the
page. Nothing else on this list is close.

**Why it's parked.**

- **SSG turns a broken demo into a broken build.** Today a demo touching `window` or
  `canvas` renders an error in one pane and the site ships. Server-rendered, it fails
  the docs build. That's a categorically worse failure mode and forces a per-demo
  opt-out — the kind of option `packages/rspress/CLAUDE.md`'s "Deliberately not
  handled" section exists to prevent.
- **Two sources of truth for demo code.** The injected import and the `files` strings
  would drift in dev, where HMR updates one and not the other.
- Externals move from the lazy virtual-module thunks into the page chunk, so they load
  eagerly and must resolve at build time.
- The reader's payload is an abstract problem at this scale.

**What it would take.** A time-boxed spike answers the only question that matters:
hand-write the injected import into `basic.mdx`, render `<Demo />` directly, build, and
see whether Rspress SSG renders it. Then repeat with a demo that touches `window` and
look at the failure. That either kills the design or de-risks its expensive half.

**Related, cheaper, and separable:** gate the first compile on the demo entering the
viewport (`IntersectionObserver` around `CodeRunner`'s mount flush, plus
`prefetchImports`). Roughly an afternoon, no fidelity risk, no SSR exposure. It only
helps readers who never scroll to the demo, so it doesn't substitute for the above —
but it's worth taking on its own terms under either architecture.

**If the static path is ever built:** show the compiler-loading overlay only on the
interaction that triggers the compile, never at first paint. The static render is a
real hydrated React component, fully interactive — an overlay at first paint would
report working content as pending. `Preview.module.css`'s `.error` block is the
existing pattern to copy.

## Tree-shaking the virtual module's externals

**Implemented and measured, then rejected.** Branch `feat/tree-shake-externals`.

`getVirtualModulesCode.ts` emits `() => import('pkg')`, which is opaque to the
bundler — it can't know which named exports the runtime will read, so it keeps the
whole barrel. Adding a `webpackExports` magic comment with the names collected at build
time fixes that:

```js
importsMap.set('@react-three/drei', () => import(/* webpackExports: ["Float","Stars"] */ '@react-three/drei'));
```

**Measured:** `goWild` 1,991,750 → 1,334,000 B (−33%). `camera-controls` (491 KB,
pulled in only by drei's barrel re-export) disappears entirely, and drei's own chunk
drops 188 KB → 17 KB. `basic` unchanged.

**Why rejected.** The entire win is one library. Only drei benefits meaningfully;
`@react-three/fiber` gives back 376 B, postprocessing 228 B, `three` can't be subset at
all because a demo uses `import * as THREE`. Against that it costs a
`Map<string, Set<string> | null>` with sticky-null merge semantics threaded through
four build files, and a real behavior regression: **editing a demo in the browser to
use an export no demo source imports now throws.** In a live-editing tool, exploring a
library's API by editing is close to the point.

**What would change the calculus.** A site where several heavy barrel-file packages are
used across many demos, or a way to widen the retained set at runtime on demand.
